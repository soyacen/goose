模块按职责分层组织：
- 传输层 `conn.go`：封装 `github.com/coder/websocket.Conn`，提供带缓冲异步写队列 (`writeCh`)、周期性 ping/pong keepalive、上下文驱动生命周期以及优雅关闭的 `Conn`，对外暴露非阻塞 `Send([]byte)` 和阻塞 `Read(ctx)`。
- 编解码层 `codec.go`：定义 `Codec` 接口（Marshal/Unmarshal），默认使用 `encoding/json` 的 `JSONCodec`，可被替换为 protobuf/msgpack 等。
- 服务契约 `service.go`：声明领域消息类型与 gRPC 风格的 `StreamServiceServer` / `StreamService` 及其泛型 stream 客户端接口（`ClientStreamingClient`、`ServerStreamingClient`、`BidiStreamingClient`），作为协议边界。
- 服务端流实现 `server_stream.go`：以组合方式实现 `ServerStream` 基类 `serverStream`（通过 `Codec` 做序列化、将 WebSocket close status 映射为 `io.EOF`），再派生 `serverClientStream`、`serverServerStream`、`serverBidiStream` 三种具体流。
- HTTP 接入层 `handler.go`：三个 `*StreamHandler` 分别对应 `/ws/client-stream`、`/ws/server-stream`、`/ws/bidi-stream`，负责 WS 升级、连接数限流、创建 `Conn` + 流并委派给 `StreamServiceServer` 的具体方法。
- 业务实现 `service_impl.go`：`streamServiceImpl` 演示三种模式的业务逻辑（聚合计数、定时推送、回显）。
- 客户端 `client.go`：生产级 resilient client，内置指数退避+抖动重连、状态机（Disconnected/Connecting/Connected/Reconnecting/Closed）、可选 OnMessage/OutgoingMessages 通道以适配三种模式。
- 入口 `main.go`：组装 `ServerConfig`、`ConnConfig`、`JSONCodec`、`StreamServiceImpl`，注册 liveness/readiness/health 探针，监听 SIGINT/SIGTERM 并按三阶段优雅关停（标记 not-ready → PreStopDrainDelay → `http.Server.Shutdown`）。
依赖方向：main → handler → server_stream → conn → codec；service_impl 仅依赖 service.go 中定义的接口，形成清晰的生成代码 vs 用户实现的分界。
